1. 인터페이스에만 문서 어노테이션 배치
    - @Operation, @ApiResponse, @Parameter 등 OpenAPI 관련 어노테이션은 컨트롤러 인터페이스에만 둔다
    - 구현체 클래스에는 순수 비즈니스 로직만 남겨서 가독성과 재사용성 확보

2. 반복되는 응답 정의는 메타(조합) 어노테이션으로 묶기
    - 자주 쓰이는 200/400/401/403/500 응답은  
      @CommonApiResponses, @AuthRequiredApi, @PagedListApi 등으로 한번에 정의
    - 개별 API 선언이 짧아져서 유지·보수가 훨씬 수월

3. 실패(에러) 케이스는 전역 예외 처리기로 한 번에 문서화
    - @ControllerAdvice + 공통 에러 DTO + @ApiResponse 조합
    - 각 컨트롤러마다 실패 응답을 반복 등록할 필요 없음

4. DTO/응답 모델에만 스키마 어노테이션 붙이기
    - Domain 모델과 API 스펙을 완전히 분리
    - 문서 수정 시 비즈니스 로직(도메인) 오염 방지

5. 프록시는 횡단 관심사(로깅·트레이싱·권한 등)용으로만 사용
    - OpenAPI 문서 생성을 프록시에 맡기지 않는다
    - Springdoc 기본 흐름(코드+어노테이션 → 문서) 그대로 유지
